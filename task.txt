Build a zero-configuration font auto-detection system for the existing Bun + Express + TypeScript + Vercel font server. The core requirement: the user should be able to drop a .woff2 (or .woff/.ttf/.otf) file into a folder, and the system automatically detects it, parses its metadata, generates a corresponding @font-face route, and makes it usable — with ZERO manual config file edits, zero restarts, zero registration steps.

## Important constraint to respect
This runs on Vercel, which has a read-only filesystem at runtime and no persistent background process. "Auto-detection" therefore CANNOT mean a live file-watcher (chokidar, fs.watch, etc.) running in production — that pattern doesn't exist in serverless. Auto-detection means: the system re-scans the fonts directory fresh on every relevant request (or at build time), so the moment a new file is committed and deployed, it's automatically picked up with no code changes. Design entirely around this model. Do NOT implement fs.watch in any code path that runs in the deployed function — only allowed in the optional local-dev-only watcher described in section 5.

## 1. Folder convention (the ONLY thing the user manually does)
- Folder: /public/fonts/
- The user drops files directly into family-named subfolders, e.g.:
  public/fonts/Inter/Inter-Regular.woff2
  public/fonts/Inter/Inter-Bold.woff2
  public/fonts/Inter/Inter-Italic.woff2
  public/fonts/Lora/Lora-Regular.woff2
- If a file is dropped directly into public/fonts/ with no subfolder (e.g. public/fonts/Roboto-Regular.woff2), the system must still auto-detect it and infer the family name by stripping known weight/style keywords and extensions from the filename (e.g. "Roboto-Regular.woff2" → family "Roboto"). Document this fallback clearly in code comments.
- No manifest file, no JSON registry, no config the user has to touch. The filesystem IS the source of truth.

## 2. Auto-detection engine (src/utils/fontScanner.ts)
Build a single function, e.g. `scanFonts(fontsDir: string): FontFamily[]`, that:

a) Recursively walks the fonts directory (one level of subfolders expected, but handle arbitrary depth gracefully — just use the immediate parent folder name as family for nested files, or the inferred name for root-level files).

b) For every file matching allowed extensions (.woff2, .woff, .ttf, .otf — case-insensitive, so .WOFF2 also works):
   - Extract family name (subfolder name, or inferred from filename per section 1)
   - Extract weight using this keyword map (case-insensitive, matched anywhere in filename):
     Thin=100, ExtraLight=200ULTRALIGHT=200, Light=300, Regular=400, Normal=400, Medium=500, SemiBold=600 DemiBold=600, Bold=700, ExtraBold=800 UltraBold=800, Black=900 Heavy=900
     Default to 400 if no keyword matches.
   - Extract style: "Italic" or "Oblique" anywhere in filename (case-insensitive) → italic, else normal.
   - Build a clean public URL for the file: `${PUBLIC_URL}/fonts/${family}/${filename}` (URL-encode the filename safely, handle spaces).
   - Determine format string for @font-face src: woff2→"woff2", woff→"woff", ttf→"truetype", otf→"opentype".

c) Group all detected files by family name, producing a structure like:
   {
     family: "Inter",
     files: [
       { weight: 400, style: "normal", format: "woff2", url: "...", filename: "Inter-Regular.woff2" },
       { weight: 700, style: "normal", format: "woff2", url: "...", filename: "Inter-Bold.woff2" }
     ]
   }

d) Must be PURE and side-effect-free — same directory contents always produce the same output. No caching layer inside this function itself (caching, if any, happens one level up — see section 3).

e) Handle edge cases without crashing:
   - Empty fonts directory → return empty array, not an error.
   - Files with no extension or wrong extension → silently skip (do not include in output, do not throw).
   - Duplicate filenames in different subfolders → both included independently, no collision (they're in different families/paths).
   - Malformed filenames that don't match any weight/style keyword → still included, with weight 400 / style normal defaults, never dropped silently. Log a console.warn so the user notices if a filename didn't parse as expected (helps them rename files to match convention if needed).

## 3. Wiring detection into the existing routes
Update the existing fonts.controller.ts to use scanFonts() as the single source of truth for:
- GET /fonts.css → iterate scanFonts() output, emit one @font-face block per file, font-display: swap on each.
- GET /fonts/list → return scanFonts() output directly as JSON (already public-URL-safe, no filesystem paths).
- GET /health → include fontsLoaded: scanFonts().reduce((sum, f) => sum + f.files.length, 0) in the response.

Since Vercel functions are stateless per-invocation but the bundled filesystem read is cheap (a few dozen files, simple fs.readdir calls), call scanFonts() fresh on every request to these routes. Do NOT add a manual in-memory cache with TTL logic — for this file count it adds complexity without meaningful benefit, and risks staleness bugs across container reuse. Keep it simple: scan every time.

## 4. "Creates a route for it" — what this actually means here
Clarify and implement this correctly: in a static + serverless architecture, you do NOT dynamically register new Express routes per font file (that's unnecessary and doesn't match how the static files are served). Instead, "auto route" means:
- Every file in public/fonts/** is automatically reachable at its direct static URL the moment it's deployed — Vercel's static serving handles this with zero code (confirm this already works via the existing vercel.json from the previous setup, do not duplicate static serving logic in Express).
- The "auto-generated route" that DOES need code is /fonts.css, which auto-includes every detected file as an @font-face block — this is the dynamic, code-driven part of "auto detection → auto route."
- Make this distinction explicit in a code comment at the top of fontScanner.ts so future-you doesn't get confused and try to build literal per-file Express routes.

## 5. Optional: local dev convenience watcher (NOT for production)
Add an OPTIONAL local-only dev script, e.g. src/dev/watch-fonts.ts, that:
- Uses Bun's native file watching (`bun --watch` already covers this for code changes, but for font file changes specifically, use a simple fs.watch on FONTS_DIR) to log to console whenever a font file is added/removed during local development, e.g. "[fonts] Detected new file: Inter/Inter-Bold.woff2 → run `bun run dev-server.ts` requests will pick this up automatically."
- This is purely a nice-to-have console notifier for local dev ergonomics — it does NOT need to trigger any rebuild, since scanFonts() already re-reads the directory fresh on every request anyway.
- Clearly comment that this file is never imported or used in api/index.ts (the production entry point) — it's a standalone dev utility only, run via an optional "watch:fonts" script in package.json.

## 6. Validation & safety
- Validate filenames before processing: reject (skip + warn) any filename containing path traversal sequences (`..`), null bytes, or unexpected characters, even though they're coming from a trusted local fonts dir — defensive coding habit.
- Validate file size is reasonable, log a console.warn if any single font file exceeds 5MB (likely a mistake, e.g. uncompressed TTF when woff2 was intended) — do not block it, just warn.
- Ensure scanFonts() never throws on a single bad file — wrap per-file processing in try/catch, skip + warn on individual failures, so one bad file doesn't break the entire fonts.css generation for every other font.

## 7. Type definitions
Define and export clean types (e.g. in src/utils/fontScanner.ts or a separate types.ts):
  type FontFile = { weight: number; style: "normal" | "italic"; format: string; url: string; filename: string };
  type FontFamily = { family: string; files: FontFile[] };
Use these types consistently across the controller, routes, and scanner — no `any` types anywhere in this feature.
